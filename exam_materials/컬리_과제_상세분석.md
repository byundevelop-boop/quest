- 기술 제약: Kotlin 사용, MockInterceptor 기반 네트워크 구성, 멀티 모듈 구조 필요.

## 2) 요구사항을 구현 단위로 재정의

## A. 화면/레이아웃
- 메인 화면은 여러 섹션으로 구성.
- 섹션 타입 3가지:
  - `vertical`: 세로 스크롤 상품 리스트
  - `horizontal`: 가로 스크롤 상품 리스트
  - `grid`: `3열 x 2행`, 화면상 6개 상품
- 각 섹션은 **섹션 제목(title)** 을 상품과 별도로 표시.
- 섹션 간 구분선 존재(첨부 UI 기준).
- pull-to-refresh 제공.

## B. 상품 카드 표현
- 카드 타입 2종:
  - `horizontal` + `grid` 공통 카드
  - `vertical` 전용 카드
- 텍스트/가격 규칙:
  - 제목
    - horizontal/grid: 2줄 + 말줄임
    - vertical: 1줄 + 말줄임
  - 할인 존재 시
    - 할인율 색상: `#fa622f`
    - 할인가: bold
    - 원가: 취소선
  - 할인 미존재 시
    - 원가만 노출
  - 가격 배치
    - horizontal/grid: 1행(할인율 + 할인가), 2행(원가)
    - vertical: 한 줄에 가격 정보 노출
- 찜하기 아이콘 상태(on/off) 반영.

## C. 데이터/네트워크
- `GET /sections?page={Int}`: 섹션 목록 + paging 정보.
- `GET /section/products?sectionId={Int}`: 해당 섹션 상품 목록.
- BaseUrl: `https://kurly.com/`
- Retrofit의 OkHttpClient에 `MockInterceptor` 추가.
- 서버 없이 찜 상태 로컬 구현.

## D. 프로젝트 구조
- 멀티 모듈 필수 (`mockserver` 모듈 연동 전제).

## 3-1. UI 이미지에서 확인된 명시 규칙

### horizontal/grid 상품 카드
- 이미지 크기: 가로 `150dp`, 세로 `200dp`
- 제목: 2줄 + 말줄임
- 할인율: 할인 있을 때만 노출, 색상 `#fa622f`
- 판매금액: bold (할인 있으면 할인가, 없으면 일반가)
- 원래가격: 할인 있을 때만 노출 + 취소선

### vertical 상품 카드
- 이미지 비율: 가로:세로 = `6:4` (너비 채움)
- 제목: 1줄 + 말줄임
- 할인율/가격 규칙은 동일

### 섹션 구성 힌트
- `Horizontal 형 섹션 (가로 스크롤 가능)`
- `Grid 형 섹션 (row 2, col 3)`
- `Vertical 형 섹션`
- `pull to refresh 새로고침 기능`
- `섹션 간 구분선`

## 3-2. MockServer 코드 기준 API 동작

`mockserver/src/main/java/com/kurly/android/mockserver/MockServer.kt`
- path 라우팅:
  - `/sections` -> `sections/sections_{page}.json` (page 미지정 시 1)
  - `/section/products` -> `section/products/section_products_{sectionId}.json`
- 파일 미존재/파라미터 누락 시 `null` 반환

`mockserver/src/main/java/com/kurly/android/mockserver/MockInterceptor.kt`
- 요청마다 지연: `1~2초 랜덤 sleep`
- `mockServer.get(...) == null` 이면 `500`, message `에러 발생`
- 정상 시 `200`, content-type `application/json`

## 4) 구현 시 요구사항별 완료 체크 기준

|요구사항|완료 체크 기준|검증 방법|
|---|---|---|
|멀티 섹션 메인|서버 데이터로 n개 섹션 렌더링|`/sections?page=1` 응답 기반 렌더링 확인|
|타입별 레이아웃|vertical/horizontal/grid가 시안 규칙대로 표시|섹션 타입별 UI 스냅샷 비교|
|섹션 타이틀|각 섹션 title이 카드들과 분리 표시|UI 확인|
|상품 타이틀 줄수|horizontal/grid=2줄, vertical=1줄 + ellipsize|긴 상품명으로 수동/스크린샷 테스트|
|가격 노출 규칙|할인 유무에 따라 텍스트/색상/취소선 조건 충족|할인/비할인 샘플 ID로 검증|
|찜하기|서버 없이 상태 토글 및 아이콘 변경|탭 이벤트/상태 복원 테스트|
|새로고침|pull-to-refresh 동작, 목록 재조회|스와이프 후 API 재호출 확인|
|페이징|next_page 기반으로 추가 섹션 로딩|페이지 1→4 로딩 및 종료 검증|
|오류 처리|500 응답 시 사용자에게 실패 상태 노출|강제 오류 요청으로 검증|

## 5) 권장 아키텍처(평가 대응형)

## 모듈
- `:app` (엔트리)
- `:mockserver` (제공 모듈, interceptor)
- 권장 추가:
  - `:core:model` (도메인/DTO 분리)
  - `:core:network` (Retrofit/OkHttp/DI)
  - `:feature:main` (메인 화면 UI/VM)

## 레이어
- Data: Retrofit API + Repository 구현
- Domain: UseCase(`LoadSections`, `RefreshSections`, `ToggleFavorite`)
- Presentation: ViewModel + UI State

## 상태 모델 예시
- `MainUiState`
  - `sections: List<SectionUiModel>`
  - `isRefreshing: Boolean`
  - `isLoadingNextPage: Boolean`
  - `nextPage: Int?`
  - `error: UiError?`

### 찜 상태 저장 전략
- 최소 요구 충족: 메모리(Map<productId, Boolean>)
- 권장: DataStore/Room으로 앱 재시작 후 복원까지 지원

## 6) 구현 난이도 포인트와 리스크

- 랜덤 네트워크 지연(1~2초)로 스크롤/로딩 UI 품질이 드러남.
- 섹션별 상품 수 가변(3~9) -> 어댑터 구조가 유연해야 함.
- 중복 상품 ID 존재 -> 찜 상태 key를 productId로 둘 경우 의도된 동기화인지 정책 필요.
- 마지막 페이지(`sections_4.json`)는 `next_page` 없음 -> 종료 조건 명확히 처리 필요.
- 500 오류 경로(잘못된 page/sectionId) 처리 없으면 UX 감점 가능.
